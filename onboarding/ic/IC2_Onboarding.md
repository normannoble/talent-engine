# 30-60-90 Day Onboarding: Software Engineer (IC2)

## Overview

**Level:** IC2 â€” Software Engineer
**Focus:** Building independence, owning features, establishing as reliable contributor
**Manager's Role:** Supportive guidance, clear context, increasing autonomy

IC2s are capable engineers who work independently on defined tasks. The goal is to quickly establish them as trusted contributors who deliver reliably.

---

## Days 1-30: Learn

### Theme: Understand the Landscape

**Primary Objectives:**
- Understand team domain, codebase architecture, and tech stack
- Build relationships with team and key stakeholders
- Complete initial meaningful contributions
- Learn team processes and norms

### Week 1: Orientation

| Day | Focus | Activities |
|-----|-------|------------|
| 1 | Welcome | Team intros, workspace setup, org context |
| 2 | Domain | Product area deep-dive, user problems, business context |
| 3 | Technical | Architecture overview, codebase structure, key systems |
| 4 | Process | Development workflow, PR standards, deployment process |
| 5 | Connect | 1:1s with teammates, meet key cross-functional partners |

**Key Meetings to Schedule:**
- [ ] Manager 1:1 (30 min, 2x/week initially)
- [ ] 1:1 with each teammate
- [ ] Product manager introduction
- [ ] Designer introduction (if applicable)
- [ ] Onboarding buddy check-ins

### Weeks 2-4: Immersion

**Learning Goals:**
- [ ] Deep understanding of team's technical domain
- [ ] Know the key systems and how they interact
- [ ] Understand current priorities and roadmap
- [ ] Identify technical debt and improvement areas
- [ ] Complete first feature or significant task

**Activities:**
- Deep-dive into 2-3 key areas of the codebase
- Review recent PRs to understand patterns and standards
- Take on a real feature (scoped appropriately)
- Participate actively in team ceremonies
- Document learnings and questions

**Technical Milestones:**
- [ ] Can work effectively in the codebase
- [ ] First feature/significant contribution shipped
- [ ] Understands testing and quality standards
- [ ] Comfortable with deployment process
- [ ] Can debug issues in team's systems

### 30-Day Check-in

**Manager Discussion Points:**
- What's your understanding of our domain and priorities?
- What patterns and practices have you noticed?
- How are relationships with the team?
- What would help you be more effective?
- Feedback on early contributions

**Success Indicators:**
- [ ] Good grasp of team's technical domain
- [ ] First meaningful contribution shipped
- [ ] Productive working relationships established
- [ ] Operating with appropriate independence
- [ ] Asking good questions, contributing in discussions

---

## Days 31-60: Contribute

### Theme: Deliver Independently

**Primary Objectives:**
- Own features end-to-end
- Deliver with minimal guidance
- Build reputation for reliability
- Start identifying improvements

### Weeks 5-6: Feature Ownership

**Activities:**
- Own a complete feature from design to deployment
- Handle ambiguity with reasonable approaches
- Write thorough tests and documentation
- Coordinate with PM/design as needed
- Provide helpful code reviews to teammates

**Growing Independence:**
- [ ] Taking ownership of outcomes, not just tasks
- [ ] Making reasonable technical decisions
- [ ] Communicating progress and blockers clearly
- [ ] Handling on-call or support responsibilities
- [ ] Contributing quality code reviews

### Weeks 7-8: Building Trust

**Activities:**
- Deliver consistently on commitments
- Take on slightly larger scope
- Start improving team processes or tooling
- Mentor IC1s if present on team
- Build relationships outside immediate team

**Expanding Impact:**
- [ ] Reliable delivery with good quality
- [ ] Starting to influence technical decisions
- [ ] Identifying and fixing issues proactively
- [ ] Helping teammates succeed
- [ ] Contributing beyond assigned work

### 60-Day Check-in

**Manager Discussion Points:**
- How is ownership going? What's working?
- Where do you want to grow technically?
- What improvements have you identified?
- How are cross-functional relationships?
- Feedback on delivery and quality

**Success Indicators:**
- [ ] Delivering features independently
- [ ] Quality and reliability established
- [ ] Making good technical decisions
- [ ] Contributing to team beyond own work
- [ ] Clear on role and expectations

---

## Days 61-90: Own

### Theme: Establish as Trusted Contributor

**Primary Objectives:**
- Full ownership of scope areas
- Consistent delivery at IC2 level
- Drive small improvements
- Set trajectory for continued growth

### Weeks 9-10: Full Ownership

**Activities:**
- Own your area with full responsibility
- Drive improvements you've identified
- Handle complexity without escalation
- Be a reliable resource for your domain
- Contribute to planning and estimation

**Demonstrating IC2 Expectations:**
- [ ] End-to-end feature delivery
- [ ] Quality code with good test coverage
- [ ] Clear communication and documentation
- [ ] Proactive problem-solving
- [ ] Growing technical judgment

### Weeks 11-12: Setting Trajectory

**Activities:**
- Look ahead to next quarter
- Set development goals
- Identify areas for IC3 growth
- Solidify relationships and reputation
- Transition fully from "new hire" to "team member"

**Preparing for Ongoing Success:**
- [ ] Development plan for IC3 growth areas
- [ ] Clear on IC2 expectations
- [ ] Established as reliable contributor
- [ ] Know where to focus for continued growth
- [ ] Fully integrated into team

### 90-Day Review

**Manager Discussion Points:**
- Overall onboarding assessment
- Key strengths demonstrated
- Areas for growth toward IC3
- Goals for next quarter
- Recognition and feedback

**Success Indicators:**
- [ ] Delivering independently at IC2 level
- [ ] Established as reliable team contributor
- [ ] Good technical judgment
- [ ] Strong working relationships
- [ ] Clear growth trajectory

## Key Relationships to Build

| Who | Why | When to Meet |
|-----|-----|--------------|
| Direct manager | Support, feedback, career growth | Weekly |
| Product manager | Feature context, priorities, collaboration | First 2 weeks, ongoing |
| Team engineers | Collaboration, knowledge sharing | First 2 weeks, ongoing |
| Senior engineers | Technical guidance, mentorship | As needed |
| Cross-functional partners | Design, QA, etc. | First month |

---

## Common IC2 Onboarding Challenges

| Challenge | Solution |
|-----------|----------|
| Ramping too slowly | Provide more focused onboarding, clear first project |
| Trying to change too much | Channel energy; prioritize learning before improving |
| Not asking for help | Check in proactively; create psychological safety |
| Underestimating work | Coach on estimation; start with buffer |
| Struggling with ambiguity | Provide more context initially; model decision-making |

## Manager Checklist

### Pre-Arrival
- [ ] Equipment and access ready
- [ ] First project identified (meaningful, appropriately scoped)
- [ ] Key 1:1s scheduled
- [ ] Team briefed on new hire
- [ ] Documentation prepared

### Week 1
- [ ] Domain and architecture overview complete
- [ ] Cross-functional introductions done
- [ ] First project kicked off
- [ ] Regular cadence established
- [ ] Questions and blockers addressed

### Ongoing
- [ ] Appropriate level of autonomy given
- [ ] Feedback provided regularly
- [ ] Growth opportunities identified
- [ ] Wins celebrated
- [ ] Challenges addressed promptly
